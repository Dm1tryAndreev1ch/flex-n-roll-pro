# Тестовые сценарии FLEX-N-ROLL PRO

В этом документе описан полный end-to-end цикл работы системы, начиная от создания лида/сообщения в Bitrix24 и заканчивая логированием метрик в Grafana.

---

## 🏗 Подготовка к тестированию

1. Убедитесь, что все контейнеры запущены (включая мониторинг):
   ```bash
   docker compose up -d
   docker compose -f docker-compose.monitoring.yml up -d
   ```
2. Откройте **Bitrix24 CRM** в браузере.
3. Откройте дашборд **Grafana** (http://localhost:3001) для наблюдения за активностью.
4. Откройте логи webhook-сервера: `docker compose logs -f webhook`

---

## Сценарий 1: Обработка нового веб-лида

**Цель:** Проверить захват лида, работу локальной нейросети (LM Studio) для классификации, назначение правильного менеджера и создание задачи со сроком (SLA).

### Шаг 1: Создание Лида в Bitrix24
1. В левом меню Bitrix24 перейдите в **CRM** → **Лиды**.
2. Нажмите **Создать** лид.
3. Заполните данные:
   * **Название лида:** `Заказ на термоусадочную этикетку`
   * **Комментарий:** `Добрый день! Нам нужно 50 000 sleeve-этикеток для новых бутылок лимонада. Размеры 150х60мм. Срок - максимум до среды, очень срочно.`
   * **Имя:** `Иван`
   * **Компания:** `ООО Напитки Плюс`
   * **Телефон:** `+375 29 111-22-33`
4. Сохраните лид.

### Шаг 2: Что происходит внутри системы
1. **Bitrix24** отправляет событие `ONCRMLEADADD` на наш `fnr-webhook`.
2. **Вебхук** получает ID лида и запрашивает его полные детали (комментарий, контакты) через REST API.
3. Собранный текст отправляется в **LM Studio** для классификации:
   * *Ожидаемый intent:* `quote_request` (запрос КП)
   * *Ожидаемый product_type:* `sleeve`
   * *Ожидаемый urgency:* `critical` (до среды, срочно)
   * *Ожидаемый приоритет:* `1` или `2`
   * *Ожидаемый роутинг:* `sales`
4. Из пула `sales` по **Round-Robin** выбирается менеджер.
5. Сервис обновляет поля лида в Bitrix24 (заполняет UF_CRM_PRODUCT_TYPE, UF_CRM_URGENCY, Priority).
6. Создается **Задача** на выбранного менеджера с дедлайном (например, +1 час для приоритета HOT).

### Шаг 3: Проверка результата
1. Обновите страницу лида в Bitrix24. Вы должны увидеть обновленные пользовательские поля и AI-комментарий.
2. Проверьте вкладку **Задачи** у лида — там должна быть новая задача с названием вида `[Лид #ID] [HOT] Запрос КП — Sleeve-этикетка`.
3. В **Grafana** на дашборде `FNR Overview` график "AI классификаций/мин" должен показать скачок для интента `quote_request`.

## Сценарий 2: AI Анализ звонка с клиентом

**Цель:** Проверить, что запись разговора, прикрепленная к сделке, автоматически транскрибируется (Whisper) и анализируется на эмоции и соблюдение скриптов (GPT).

### Шаг 1: Добавление звонка
1. Перейдите в **CRM** → **Сделки**.
2. Создайте или откройте существующую сделку.
3. В таймлайне сделки добавьте **Дело -> Звонок** (в реальной среде звонки добавляются интеграцией телефонии, например Voximplant). Обязательно прикрепите аудиозапись `.mp3` или `.wav`.
4. Завершите звонок.

### Шаг 2: Что происходит внутри системы
1. Сервис `fnr-commanalysis` получает событие `ONVOXIMPLANTCALLEND` (или вы можете дернуть эндпоинт `/api/calls/analyze-by-deal` вручную).
2. Скачивается `.mp3` файл из Bitrix24 во временную папку.
3. **Whisper** транскрибирует аудио в текст.
4. Текст отправляется в **OpenAI GPT-4o** для извлечения метрик: корректность скрипта, тон (sentiment), выявление возражений.
5. `fnr-commanalysis` генерирует красивую HTML-карточку.
6. В сделке обновляются поля `UF_TRANSCRIPT` и `UF_CALL_SCORE`.
7. HTML-карточка публикуется как новый комментарий в таймлайне сделки.

### Шаг 3: Проверка результата
1. В таймлайне сделки проверьте появление комментария с оценкой: 
   * `Оценка звонка: 85/100` 
   * Тональность
   * Замечания AI
2. В **Grafana** счетчик "CommAnalysis — анализов/мин" увеличится.

## Сценарий 3: Запрос в Telegram Боте

**Цель:** Проверить, что клиент может узнать статус своего заказа (стадию сделки) просто написав номер в Telegram.

### Шаг 1: Вызов бота
1. Откройте Telegram и найдите вашего `@fnr_status_bot`.
2. Нажмите `/start`. Бот должен поприветствовать вас и показать меню.
3. Нажмите кнопку **📦 Проверить заказ**.

### Шаг 2: Проверка номера и контактов
1. Бот спросит номер заказа. Введите ID сделки из Bitrix24 (например, `152`).
2. Для безопасности бот спросит подтверждение (Email или Телефон). Введите телефон, который указан в контакте, привязанном к сделке.

### Шаг 3: Что происходит внутри системы
1. Бот обращается к `calculator-api` (или напрямую к битриксовому сервису) по REST: запрашивает сделку `152` и контакт.
2. Проверяет совпадение контактных данных.
3. Мапит ID стадии сделки (например, `C3:PREPARATION`) на человекопонятный текст: *"Ваш заказ находится на стадии: Подготовка к печати"*.
4. Возвращает ответ клиенту.

## Сценарий 4: Генерация кодов "Честный ЗНАК"

**Цель:** Проверить передачу состава заказа (GTIN) в ГИС МТ (MDLP) и возврат массива DataMatrix-кодов.

### Шаг 1: Вызов API (эмуляция ERP/1C)
Т.к. маркировка требует вызова извне (или кнопки в интерфейсе), мы симулируем POST-запрос к нашему микросервису:
```bash
curl -X POST http://localhost:3000/api/marking/order \
  -H "Content-Type: application/json" \
  -d '{
    "dealId": "402",
    "gtin": "04601234567890",
    "quantity": 100,
    "serialType": "AUTO"
  }'
```

### Шаг 2: Что происходит внутри системы
1. `fnr-marking` делает авторизацию в ГИС МТ через ключи в `.env`.
2. Запрашивает массив кодов (эмиссия КМ) для указанного GTIN в количестве 100 шт.
3. Сохраняет коды в свою встраиваемую БД: `/app/data/codes_db.json`.
4. Генерирует PDF-документ или CSV с кодами и сохраняет в: `/app/data/production/orders/402_codes.csv`.
5. Обновляет сделку 402 в Bitrix24: загружает файл и/или ставит чекбокс `UF_MARKING_DONE`.

### Шаг 3: Проверка результата
1. В логах маркировки `docker compose logs marking` видно успешное получение кодов.
2. В **Grafana** (панель Маркировка) счетчик показывает +100 к сгенерированным кодам.
3. В сделке Bitrix24 появился прикрепленный отчет с кодами маркировки.
